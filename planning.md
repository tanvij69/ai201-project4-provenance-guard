# Provenance Guard - Planning

## Project Overview

Provenance Guard is a backend API that analyzes submitted text and estimates whether it is more likely to have been written by a human or generated by AI. Instead of making a simple yes/no decision, the system combines multiple detection signals to produce a confidence score, displays a transparency label for users, records every decision in an audit log, and allows creators to appeal classifications they believe are incorrect.

---

# Architecture Narrative

When a creator submits a piece of text, it is sent to the `POST /submit` endpoint. The request first passes through rate limiting (Flask-Limiter), which rejects the submission if the creator has made too many requests recently.

If it passes, the submission goes through two independent detection signals. The first uses the Groq Llama model to classify whether the writing appears AI-generated or human-written based on overall writing style, coherence, and semantic patterns. The second calculates stylometric features such as sentence length variation, vocabulary diversity (type-token ratio), and punctuation usage to measure how naturally the writing varies.

The outputs of both signals are combined into a single confidence score by the confidence scoring engine. Based on this score, the system generates one of three transparency labels:

- High Confidence AI
- High Confidence Human
- Uncertain

The original text, individual signal scores, combined confidence score, transparency label, and timestamp are saved in the audit log before the API returns the response to the user.

If the creator disagrees with the result, they can submit an appeal using the `POST /appeal/<submission_id>` endpoint. Their explanation is stored alongside the original decision, the submission status changes to **Under Review**, and the appeal is added to the audit log. No automatic re-classification happens — a human is expected to review the appeal afterward.

# Detection Signals

## Signal 1: LLM-Based Classification (Groq)

### What it measures
Uses a large language model to determine whether the overall writing style resembles AI-generated or human-written text, based on semantic and stylistic coherence.

### Why it works
Large language models recognize patterns in grammar, organization, tone, and semantic consistency that often distinguish AI-generated writing from human writing, learned from exposure to large amounts of both kinds of text.

### Blind spots
- Can incorrectly classify polished or unusual human writing as AI.
- May struggle with heavily edited AI content.
- Depends on the quality of the prompt and model output.
- Acts as a black box — there is no clear explanation for why it assigned a particular score.

---

## Signal 2: Stylometric Heuristics

### What it measures
- Sentence length variation
- Vocabulary diversity (Type-Token Ratio)
- Punctuation density
- Average sentence complexity

### Why it works
Human writing tends to be more varied and inconsistent, while AI writing often produces more uniform sentence structures and vocabulary usage. This is computable directly in pure Python with no external dependencies.

### Blind spots
- Skilled or constrained writers (e.g., minimalist poets, non-native speakers) may naturally write consistently and score as "AI-like" despite being human.
- AI text can be edited afterward to appear more naturally varied.
- Short pieces of text provide fewer useful statistics to measure.

These two signals are genuinely independent: one is semantic (LLM judgment), the other is structural (statistical measurement). Combining them is more informative than either alone.

---

# Confidence Scoring

Each detection signal produces its own score. The final confidence score is a weighted combination of both signals.

Example ranges:

| Confidence | Result |
|------------|--------|
| 0.90 - 1.00 | High Confidence AI |
| 0.60 - 0.89 | Likely AI |
| 0.40 - 0.59 | Uncertain |
| 0.10 - 0.39 | Likely Human |
| 0.00 - 0.09 | High Confidence Human |

The system intentionally favors uncertainty over making incorrect accusations, since a false positive (labeling a human's work as AI-generated) is worse than a false negative on a creative writing platform. Borderline scores should produce the "Uncertain" label instead of confidently labeling human work as AI-generated.

---

# False Positive Scenario

A human poet submits a tightly controlled, minimalist poem. Their natural style has low sentence-length variance, which scores "AI-like" on the stylometric signal, while the Groq signal is uncertain on its own. The combined confidence comes out around 0.55, which falls into the "Uncertain" band rather than "High Confidence AI."

Because of this, the transparency label tells the reader that the system could not confidently determine the source, rather than issuing a damning AI accusation. The poet can then call `POST /appeal/<submission_id>`, explain that this is their natural style, and the submission's status flips to "Under Review" for a human to look at. This scenario is why the confidence band matters more than a binary yes/no: a 0.55 should visibly read differently than a 0.95.

# API Endpoints

## POST /submit

### Input
```json
{
  "text": "Submitted writing...",
  "creator_id": "creator_123"
}
```

### Returns
```json
{
  "submission_id": 1,
  "classification": "AI",
  "confidence": 0.91,
  "label": "High Confidence AI",
  "signals": {
    "llm_score": 0.93,
    "stylometric_score": 0.89
  }
}
```

---

## POST /appeal/<submission_id>

### Input
```json
{
  "reason": "I wrote this myself."
}
```

### Returns
```json
{
  "submission_id": 1,
  "status": "Under Review",
  "appeal_logged": true
}
```

---

## GET /status/<submission_id>

### Returns
```json
{
  "submission_id": 1,
  "classification": "AI",
  "confidence": 0.91,
  "status": "Under Review"
}
```

---

## GET /log

Returns all audit log entries including:

- submission ID
- timestamp
- individual signal scores
- combined confidence score
- final classification
- transparency label
- appeal status
